## 模板要求

**此技能使用模板进行LBO建模。始终首先检查是否有附加的模板文件。**

在开始任何LBO模型之前：
1. **如果附加/提供了模板文件**：完全使用该模板的结构 - 复制它并用用户的数据填充
2. **如果没有附加模板**：询问用户：*"您是否有特定的LBO模板希望我使用？如果没有，我可以使用标准模板，包括资金来源与使用、运营模型、债务计划和回报分析。"*
3. **如果使用标准模板**：复制`examples/LBO_Model.xlsx`作为起点，并用用户的假设填充

**重要**：当附加了`LBO_Model.xlsx`等文件时，您必须使用它作为模板 - 不要从头开始构建。即使模板看起来复杂或具有比需要更多的功能，也要复制它并适应用户的要求。当提供模板时，绝不要决定"从头开始构建"。

## CLAUDE的关键说明 - 请先阅读

### 环境：Office JS vs Python

**在Excel中运行（Office插件/Office JS环境）：**
- 直接使用Office JS（`Excel.run(async (context) => {...})`）- 不要使用Python/openpyxl
- 通过 `range.formulas = [["=B5*B6"]]` 编写公式 - Office JS公式在实时工作簿中原生重新计算
- 公式优于硬编码的规则同样适用：设置 `range.formulas`，任何应该是计算的都绝不设置 `range.values`
- 使用 `range.format.font.color` / `range.format.fill.color` 进行蓝色/黑色/紫色/绿色约定
- 不需要单独的重新计算步骤 - Excel原生处理计算
- **合并单元格陷阱**：不要调用 `.merge()` 然后在合并范围上设置 `.values`（抛出 `InvalidArgument` - 范围仍报告原始维度）。而是：仅在左上角单元格写入值（`ws.getRange("A7").values = [["SOURCES & USES"]]`），然后合并+格式化整个范围（`ws.getRange("A7:F7").merge(); ws.getRange("A7:F7").format.fill.color = "#1F4E79";`）

**生成独立的.xlsx文件（没有实时Excel会话）：**
- 使用如下所述的Python/openpyxl
- 编写公式字符串（`ws["D20"] = "=B5*B6"`），然后在交付前运行 `recalc.py`

本技能的其余部分使用openpyxl示例编写，但相同的原则适用于Office JS - 只需转换API调用。

### 核心原则
* **每个计算必须是Excel公式** - 绝不要在Python中计算值并将结果硬编码到单元格中。使用openpyxl时，编写 `cell.value = "=B5*B6"`（公式字符串），而不是 `cell.value = 1250`（计算结果）。模型必须是动态的，并在输入更改时更新。
* **使用模板结构** - 遵循`examples/LBO_Model.xlsx`或用户提供模板中的组织。不要发明自己的布局。
* **使用正确的单元格引用** - 所有公式应引用适当的单元格。永远不要输入应来自其他单元格的数字。
* **保持符号约定一致** - 遵循模板使用的任何符号约定（有些使用负数表示流出，有些使用正数）。始终保持一致。
* **逐部分工作，在每步与用户验证** - 完全完成一个部分，向用户展示构建的内容，运行该部分的验证检查，并在移动到下一部分之前获得确认。不要端到端构建整个模型然后展示 - 后面的部分依赖前面的部分，所以在回报已经构建后发现资金来源与使用中的错误意味着到处都要返工。

### 公式颜色约定
* **蓝色（0000FF）**：硬编码输入 - 不引用其他单元格的输入数字
* **黑色（000000）**：带计算的公式 - 任何使用运算符或函数的公式（`=B4*B5`、`=SUM()`、`=-MAX(0,B4)`）
* **紫色（800080）**：链接到**同一选项卡**上的单元格 - 没有计算的直接引用（`=B9`、`=B45`）
* **绿色（008000）**：链接到**不同选项卡**上的单元格 - 跨选项卡引用（`=Assumptions!B5`、`='Operating Model'!C10`）

### 填充色调色板 - 专业蓝/灰色调（默认，除非用户/模板另有说明）
* **保持最小化** - 仅使用蓝色和灰色进行单元格填充。不要引入绿色、黄色、红色或多种强调色。专业的LBO模型使用克制。
* **默认填充调色板：**
  * **部分标题**（资金来源与使用、运营模型等）：深蓝色 `#1F4E79` 配白色粗体文本
  * **列标题**（第1年、第2年等）：浅蓝色 `#D9E1F2` 配黑色粗体文本
  * **输入单元格**：浅灰色 `#F2F2F2`（或仅白色）- 蓝色*字体*是信号，填充是次要的
  * **公式/计算单元格**：白色，无填充
  * **关键输出**（IRR、MOIC、退出股权）：中等蓝色 `#BDD7EE` 配黑色粗体文本
* **这就是整个调色板。** 3种蓝色 + 1种灰色 + 白色。如果模板使用自己的颜色，则遵循模板。
* 注意：上面提到的蓝色/黑色/紫色/绿色**字体**颜色用于区分输入vs公式vs链接。这些与此处的**填充**调色板分开 - 两者协同工作。

### 数字格式标准
* **货币**：`$#,##0;($#,##0);"-"` 或 `$#,##0.0`，具体取决于模板
* **百分比**：`0.0%`（一位小数）
* **倍数**：`0.0"x"`（一位小数）
* **MOIC/详细比率**：`0.00"x"`（两位小数以提高精度）
* **所有数字单元格**：右对齐

## 模板分析阶段 - 首先执行此操作

在填充任何公式之前，彻底检查模板：

1. **映射结构** - 识别每个部分的位置以及它们如何相互关联。注意哪些部分流入其他部分。

2. **理解时间线** - 哪些列代表哪些期间？是否有"收盘"或"形式上"列？预测期从哪里开始？

3. **识别输入vs公式单元格** - 模板通常使用颜色编码、边框或阴影来指示哪些单元格需要输入vs公式。尊重这些约定。

4. **仔细阅读现有标签** - 行标签确切地告诉您期望什么计算。不要假设 - 阅读模板要求的内容。

5. **检查现有公式** - 有些模板部分填充。除非特别要求，否则不要覆盖工作公式。

6. **注意模板特定的约定** - 符号约定、小计结构、各部分如何组织、不同组件是否有单独的选项卡等。

## 填充公式 - 通用方法

对于每个需要公式的单元格，遵循此层次结构：

### 步骤1：检查模板
* 单元格是否已经有公式？如果是，验证它是正确的然后继续。
* 是否有注释或注释指示预期的计算？
* 行/列标签是否使计算显而易见？
* 相邻单元格是否显示您应该遵循的模式？

### 步骤2：检查用户的说明
* 用户是否指定了特定的计算方法？
* 是否有影响此公式的陈述假设？
* 是否提到了任何特殊要求？

### 步骤3：应用标准做法
* 如果模板和用户都没有指定，使用标准LBO建模约定
* 记录您所做的任何假设
* 如果真的不确定，请询问用户

## 常见问题区域

以下计算模式在LBO模型中经常引起问题。遇到这些时要特别注意：

### 平衡部分
* 当两个部分必须相等时（例如，资金来源 = 使用），通常有一个项目是"插件"（平衡数字）
* 识别哪个项目是插件并将其计算为差额

### 税务计算
* 税务公式应仅引用相关收入行和税率
* 不应引用不相关的部分（例如，债务计划）
* 考虑损失是否创建税收盾或只是被忽略

### 利息和循环引用
* 利息计算可能创建循环性，如果它们引用受现金流影响的余额
* 使用**期初余额**（不是平均或期末）来打破循环引用
* 模式：利息 → 现金流 → 偿还 → 期末余额（如果利息使用期末余额，这会循环回来）

### 债务偿还/现金归集
* 当存在多个债务层级时，通常有一个优先顺序
* 现金归集应遵循优先级瀑布
* 余额不能为负 - 适当使用MAX或MIN函数

### 回报计算（IRR/MOIC）
* 现金流必须具有正确的符号：投资 = 负，收益 = 正
* 如果使用XIRR，需要相应的日期
* 如果使用IRR，现金流应在连续期间内
* MOIC = 总收益 / 总投资

### 敏感性表
* **使用奇数维度**（5×5或7×7）- 永远不要4×4或6×6。奇数维度保证真正的中心单元格。
* **中心单元格 = 基准情景。**围绕模型的实际假设对称地构建行和列轴值（例如，如果基准退出倍数 = 10.0x，轴 = `[8.0x, 9.0x, 10.0x, 11.0x, 12.0x]`）。中心单元格的IRR/MOIC必须然后等于模型的实际IRR/MOIC输出 - 这证明表正确连接。
* **突出显示中心单元格** - 中等蓝色填充（`#BDD7EE`）+ 粗体字体，以便基准情景在视觉上锚定。
* Excel的DATA TABLE功能可能不适用于openpyxl - 相反，编写引用行/列标题的显式公式
* 每个单元格应显示不同的值 - 如果都相同，则公式没有正确变化
* 使用混合引用（例如，`$A5`用于行输入，`B$4`用于列输入）

## 验证检查清单 - 完成后运行

### 运行公式验证
```bash
python /mnt/skills/public/xlsx/recalc.py model.xlsx
```
必须返回成功，零错误。

### 部分平衡
- [ ] 任何必须平衡的部分（资金来源/使用、资产/负债）完全平衡
- [ ] 插件项目正确计算为平衡数字
- [ ] 跨部分应该匹配的金额一致

### 收入/运营预测
- [ ] 收入/顶行从驱动因素或增长率正确构建
- [ ] 所有成本和费用项目适当计算
- [ ] 小计和总计正确求和
- [ ] 利润率和比率合理
- [ ] 链接到假设正确

### 资产负债表（如适用）
- [ ] 资产 = 负债 + 权益（必须平衡）
- [ ] 所有项目链接到适当的计划或滚动结转
- [ ] 期初余额 = 前一期期末余额
- [ ] 包括检查行并显示零

### 现金流（如适用）
- [ ] 以正确的收入数字开始
- [ ] 非现金项目适当添加/减去
- [ ] 营运资本变化具有正确的符号
- [ ] 期末现金 = 期初现金 + 净现金流
- [ ] 现金余额在报表间一致

### 支持计划
- [ ] 滚动结转计划平衡（期初 + 变化 = 期末）
- [ ] 计划正确链接到主报表
- [ ] 计算项目使用适当的驱动因素
- [ ] 所有期间一致计算

### 债务/融资计划（如适用）
- [ ] 期初余额联系到来源或前一期
- [ ] 利息在适当的余额上计算（通常为期初）
- [ ] 偿还尊重现金可用性和优先级
- [ ] 期末余额不能为负
- [ ] 总计正确求和各层级

### 回报/输出分析
- [ ] 退出/终值正确计算
- [ ] 包括所有相关调整
- [ ] 现金流符号正确（投资为负，收益为正）
- [ ] IRR/MOIC公式引用完整范围
- [ ] 结果对情景合理

### 敏感性表（如适用）
- [ ] 网格维度为奇数（5×5或7×7）- 存在真正的中心单元格
- [ ] 行和列轴值围绕基准情景对称（`[基准-2Δ，基准-Δ，基准，基准+Δ，基准+2Δ]`）
- [ ] 中心单元格输出等于模型的实际IRR/MOIC - 确认表正确连接
- [ ] 中心单元格突出显示（中等蓝色填充 `#BDD7EE`，粗体字体）
- [ ] 行和列标题包含适当的输入值
- [ ] 每个数据单元格包含公式（不是硬编码）
- [ ] 每个数据单元格显示不同的值
- [ ] 值按预期方向移动（更高的退出倍数 → 更高的IRR等）

### 格式
- [ ] 硬编码输入为蓝色（0000FF）
- [ ] 计算的公式为黑色（000000）
- [ ] 同选项卡链接为紫色（800080）
- [ ] 跨选项卡链接为绿色（008000）
- [ ] 所有数字右对齐
- [ ] 适当应用数字格式
- [ ] 没有单元格显示错误值（#REF!、#DIV/0!、#VALUE!、#NAME?）

### 逻辑合理性检查
- [ ] 数字是合理的数量级
- [ ] 趋势合理（增长、下降、如预期稳定）
- [ ] 没有明显错误的值（应为正数时为负，不可能的百分比等）
- [ ] 关键输出在分析类型的合理范围内

## 避免的常见错误

| 错误 | 出什么问题 | 如何修复 |
|-------|-----------------|------------|
| 硬编码计算值 | 输入更改时模型不更新 | 始终使用引用源单元格的公式 |
| 复制后单元格引用错误 | 公式指向错误的单元格 | 验证所有链接，使用适当的$锚定 |
| 循环引用错误 | 模型无法计算 | 对利息类计算使用期初余额，打破循环 |
| 部分不平衡 | 应该匹配的总计不匹配 | 确保一个项目是插件（计算为差额） |
| 不可能的地方出现负余额 | 支付/使用超过可用 | 适当使用MAX(0, ...)或MIN函数 |
| IRR/回报错误 | 符号错误或范围不完整 | 检查现金流符号并确保公式覆盖所有期间 |
| 敏感性表显示相同值 | 公式不随输入变化 | 检查单元格引用 - 需要混合引用（$A5，B$4） |
| 滚动结转不一致 | 期初 ≠ 前一期期末 | 验证期间之间的链接 |
| 符号约定不一致 | 加法变成减法或反之 | 始终如一地遵循模板的约定 |

## 与用户合作 - 逐部分检查点

* **如果模板结构不清楚**，请继续之前询问
* **如果用户的要求与模板冲突**，确认他们的偏好
* **完成每个主要部分后**，停止并在继续之前验证用户：
  - **资金来源与使用之后** → 显示平衡的表，确认插件正确，在构建运营模型之前获得批准
  - **运营模型/预测之后** → 显示预测的P&L，确认增长率和利润率看起来正确，在债务计划之前获得批准
  - **债务计划之后** → 显示期初/期末余额和利息，确认瀑布逻辑，在回报之前获得批准
  - **回报（IRR/MOIC）之后** → 显示现金流系列和输出，确认符号和范围，在敏感性表之前获得批准
  - **敏感性表之后** → 显示每个单元格变化，确认基准情景落在预期位置
* **如果在验证期间发现错误**，在移动到下一部分之前修复它们
* **展示您的工作** - 在有帮助时解释关键公式或假设
* **从未在没有检查每个部分的情况下展示完成的模型** - 在源头发现错误的单元格引用比从损坏的IRR向后追踪更快
