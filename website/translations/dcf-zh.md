# DCF模型构建器

## 概述

本技能按照投资银行标准创建机构质量的DCF模型进行股权估值。每个分析生成一个详细的Excel模型（敏感性分析包含在DCF表的底部）。

## 工具

- 默认使用用户提供的所有信息和可用于数据源的MCP服务器。

## 关键约束 - 请先阅读

这些约束适用于所有DCF模型构建。在开始之前审查：

**环境：Office JS vs Python/openpyxl：**
- **在Excel中运行（Office插件/Office JS环境）：** 直接使用Office JS - 不要使用Python/openpyxl。通过 `range.formulas = [["=D19*(1+$B$8)"]]` 编写公式。不需要单独的重新计算步骤；Excel原生计算。使用 `range.format.*` 进行样式设置。公式优于硬编码的规则同样适用：设置 `.formulas`，派生单元格永远不要设置 `.values`。
- **生成独立的.xlsx文件（没有实时Excel会话）：** 使用如下所述的Python/openpyxl，然后在交付前运行 `recalc.py`。
- 本技能的其余部分使用openpyxl示例 - 在该环境中转换为Office JS API调用，但所有原则（公式字符串、单元格注释、部分检查点、敏感性表循环）同样适用。

**⚠️ Office JS合并单元格陷阱：** 当使用合并单元格构建部分标题时，不要调用 `.merge()` 然后在合并范围上设置 `.values` - Office JS仍然报告范围的原始维度并将抛出 `InvalidArgument: 输入数组中的行数或列数与范围的大小或维度不匹配`。而是，仅在左上角单元格写入值，然后合并并格式化整个范围：

```js
// 错误 - 抛出InvalidArgument：
const hdr = ws.getRange("A7:H7");
hdr.merge();
hdr.values = [["MARKET DATA & KEY INPUTS"]];  // 1×1数组 vs 1×8范围 → 失败

// 正确 - 先在单个单元格上写入值，然后合并+格式化范围：
ws.getRange("A7").values = [["MARKET DATA & KEY INPUTS"]];
const hdr = ws.getRange("A7:H7");
hdr.merge();
hdr.format.fill.color = "#1F4E79";
hdr.format.font.bold = true;
hdr.format.font.color = "#FFFFFF";
```

这适用于DCF中的每个合并部分标题（市场数据、情景块、现金流预测、终值、估值摘要、敏感性表）。

**公式优于硬编码（不可协商）：**
- 每个预测、利润率、折现因子、现值和敏感性单元格必须是实时的Excel公式 - 绝不要在Python中计算并写入为数字的值
- 使用openpyxl时：`ws["D20"] = "=D19*(1+$B$8)"` 是正确的；`ws["D20"] = calculated_revenue` 是错误的
- 唯一允许的硬编码数字是：(1)原始历史输入，(2)假设驱动因素（增长率、WACC输入、终值g），(3)当前市场数据（股价、债务余额）
- 如果你发现在Python中计算某物并写入结果 - 停止。当用户更改假设时，模型必须能够灵活调整。

**逐步与用户验证（不要端到端构建）：**
- 数据检索后 → 向用户显示原始输入块（收入、利润率、股份、净债务）并在预测前确认
- 收入预测后 → 向用户显示预测的顶行和增长率，在构建利润率之前确认
- FCF构建后 → 向用户显示完整的FCF时间表，在计算WACC之前确认逻辑
- WACC后 → 向用户显示计算和输入，在折现前确认
- 终值 + 现值后 → 向用户显示股权桥（EV → 股权价值 → 每股），在敏感性表之前确认
- 在每个阶段捕获错误 - 在构建敏感性表后发现错误的利润率假设意味着重建下游的所有内容

**敏感性表：**
- **使用奇数行和列**（标准：5×5，有时7×7）- 这保证了一个真正的中心单元格
- **中心单元格 = 基准情景。** 构建轴值，使中间行标题和中间列标题完全等于模型的实际假设（例如，如果基准WACC = 9.0%，中间行为9.0%；如果终值g = 3.0%，中间列为3.0%）。中心单元格的输出必须因此等于模型的实际隐含股价 - 这是敏感性表构建正确的合理性检查。
- **突出显示中心单元格**，使用中等蓝色填充（`#BDD7EE`）+ 粗体字体，以便立即可见哪个单元格是基准情景。
- 使用完整的DCF重新计算公式填充所有单元格（通常3个表 × 25个单元格 = 75）
- 使用openpyxl循环（或Office JS循环）以编程方式编写公式
- 没有占位符文本，没有线性近似，不需要手动步骤
- 每个单元格必须为该假设组合重新计算完整的DCF

**单元格注释：**
- 在创建每个硬编码值时添加单元格注释
- 格式："来源：[系统/文档]，[日期]，[参考]，[适用时的URL]"
- 每个蓝色输入必须在移动到下一部分之前有注释
- 不要推迟到最后或写"TODO：添加来源"

**模型布局规划：**
- 在编写任何公式之前定义所有部分行位置
- 首先编写所有标题和标签
- 其次编写所有部分分隔符和空行
- 然后使用锁定的行位置编写公式
- 创建后立即测试公式

**公式重新计算：**
- 在交付前运行 `python recalc.py model.xlsx 30`
- 修复所有错误直到状态为"success"
- 零公式错误要求（#REF!、#DIV/0!、#VALUE!等）

**情景块：**
- 为Bear/Base/Bull情景创建单独的块
- 在每个块内横向显示跨预测年度的假设
- 使用IF公式：`=IF($B$6=1,[Bear单元格],IF($B$6=2,[Base单元格],[Bull单元格]))`
- 验证公式引用正确的情景块单元格

## DCF流程工作流

### 步骤1：数据检索和验证

从MCP服务器、用户提供的数据和网络获取数据。

**数据源优先级：**
1. **MCP服务器**（如果配置）- 来自Daloopa等提供商的结构化金融数据
2. **用户提供的数据** - 其研究中的历史财务数据
3. **网络搜索/获取** - 需要时的当前价格、贝塔、债务和现金

**验证清单：**
- 验证净债务与净现金（对估值至关重要）
- 确认摊薄后流通股数（检查最近的回购/发行）
- 验证历史利润率与业务模型一致
- 与行业基准交叉检查收入增长率
- 验证税率合理（通常21-28%）

### 步骤2：历史分析（3-5年）

分析并记录：
- **收入增长趋势**：计算CAGR，识别驱动因素
- **利润率进展**：跟踪毛利率、EBIT利润率、FCF利润率
- **资本密集度**：D&A和CapEx占收入百分比
- **营运资本效率**：NWC变化占收入增长百分比
- **回报指标**：ROIC、ROE趋势

创建显示的汇总表：
```
历史指标（LTM）：
收入：$X百万
收入增长：X% CAGR
毛利率：X%
EBIT利润率：X%
D&A占收入百分比：X%
CapEx占收入百分比：X%
FCF利润率：X%
```

### 步骤3：构建收入预测

**方法论：**
1. 从最新的实际收入开始（LTM或最近财政年度）
2. 为每个预测年度应用增长率
3. 同时显示美元金额和计算的增长%

**增长率框架：**
- 第1-2年：反映近期可见性的较高增长
- 第3-4年：逐渐向行业平均水平温和
- 第5年以上：接近终值增长率

**公式结构：**
- 收入(第N年) = 收入(第N-1年) × (1 + 增长率)
- 增长%(第N年) = 收入(第N年) / 收入(第N-1年) - 1

**三情景方法：**
```
悲观情景：保守增长（例如，8-12%）
基准情景：最可能的情景（例如，12-16%）
乐观情景：乐观增长（例如，16-20%）
```

### 步骤4：运营费用建模

**固定/可变成本分析：**

运营费用应建模现实的运营杠杆：
- **销售和营销**：通常占收入的15-40%，具体取决于业务模式
- **研发**：科技公司通常为10-30%
- **一般和管理**：通常占收入的8-15%，随着公司扩展显示杠杆

**关键原则：**
- 所有百分比基于收入，而不是毛利润
- 建模运营杠杆：随着收入扩展，百分比应下降
- 为S&M、R&D、G&A保持单独的行项目
- 计算EBIT = 毛利润 - 总运营费用

**利润率扩展框架：**
```
当前状态 → 目标状态（第5年）
毛利率：X% → Y%（基于规模、效率证明）
EBIT利润率：X% → Y%（收入增长 + 运营费用的结果）
```

### 步骤5：自由现金流计算

**按正确顺序构建FCF：**

```
EBIT
(-) 税收（EBIT × 税率）
= NOPAT（税后净营业利润）
(+) D&A（非现金费用，占收入百分比）
(-) CapEx（占收入百分比，通常4-8%）
(-) Δ NWC（营运资本变化）
= 无杠杆自由现金流
```

**营运资本建模：**
- 计算为收入变化百分比（增量收入）
- 典型范围：收入变化的-2%至+2%
- 负数 = 现金来源（营运资本释放）
- 正数 = 现金使用（营运资本建立）

**维护性 vs 增长性CapEx：**
- 维护性CapEx：维持当前运营（约占收入2-3%）
- 增长性CapEx：支持扩张（额外占收入2-5%）
- 总CapEx应与公司的增长战略一致

### 步骤6：资本成本（WACC）研究

**股权成本的CAPM方法：**

```
股权成本 = 无风险利率 + 贝塔 × 股权风险溢价

其中：
- 无风险利率 = 当前10年期国债收益率
- 贝塔 = 5年月度股票贝塔 vs 市场指数
- 股权风险溢价 = 5.0-6.0%（市场标准）
```

**债务成本计算：**

```
税后债务成本 = 税前债务成本 × (1 - 税率)

从以下确定税前债务成本：
- 信用评级（如果可用）
- 公司债券的当前收益率
- 财务数据中的利息费用 / 总债务
```

**资本结构权重：**

```
股权市场价值 = 当前股价 × 流通股数
净债务 = 总债务 - 现金及等价物
企业价值 = 市值 + 净债务

股权权重 = 市值 / 企业价值
债务权重 = 净债务 / 企业价值

WACC = （股权成本 × 股权权重）+（税后债务成本 × 债务权重）
```

**特殊情况：**
- **净现金头寸**：如果现金 > 债务，净债务为负数
  - 债务权重可能为负
  - WACC计算相应调整
- **无债务**：WACC = 股权成本

**典型WACC范围：**
- 大盘、稳定：7-9%
- 成长型公司：9-12%
- 高增长/风险：12-15%

### 步骤7：折现率应用（5-10年预测）

**年中惯例：**
- 假设现金流在年中发生
- 折现期：0.5、1.5、2.5、3.5、4.5等
- 折现因子 = 1 / (1 + WACC)^期

**现值计算：**
```
对于每个预测年度：
FCF的现值 = 无杠杆FCF × 折现因子

示例（第1年）：
FCF = $1,000
WACC = 10%
期 = 0.5
折现因子 = 1 / (1.10)^0.5 = 0.9535
现值 = $1,000 × 0.9535 = $954
```

**预测期选择：**
- **5年**：大多数分析的标准
- **7-10年**：具有较长跑道的高增长公司
- **3年**：成熟、稳定的业务

### 步骤8：终值计算

**永续增长方法（首选）：**

```
终值FCF = 最后一年FCF × (1 + 终值增长率)
终值 = 终值FCF / (WACC - 终值增长率)

关键约束：终值增长 < WACC（否则无限价值）
```

**终值增长率选择：**
- 保守：2.0-2.5%（GDP增长率）
- 中等：2.5-3.5%
- 激进：3.5-5.0%（仅用于市场领导者）

**不要超过**：无风险利率或长期GDP增长

**退出倍数方法（替代）：**
```
终值 = 最后一年EBITDA × 退出倍数

退出倍数来自：
- 行业可比交易倍数
- 先例交易倍数
- 典型范围：8-15x EBITDA
```

**终值的现值：**
```
终值的现值 = 终值 / (1 + WACC)^最终期

其中最终期考虑时间：
5年模型使用年中惯例：期 = 4.5
```

**终值合理性检查：**
- 应占企业价值的50-70%
- 如果>75%，模型可能过度依赖终值假设
- 如果<40%，检查终值假设是否过于保守

### 步骤9：企业价值到股权价值桥

**估值摘要结构：**

```
(+) 预测FCF的现值之和 = $X百万
(+) 终值的现值 = $Y百万
= 企业价值 = $Z百万

(-) 净债务[如果为负则为+净现金] = $A百万
= 股权价值 = $B百万

÷ 摊薄后流通股数 = C百万股
= 隐含每股价格 = $XX.XX

当前股价 = $YY.YY
隐含回报 = （隐含价格 / 当前价格）- 1 = XX%
```

**关键调整：**
- **净债务 = 总债务 - 现金及等价物**
  - 如果为正：从EV中减去（减少股权价值）
  - 如果为负（净现金）：加到EV（增加股权价值）
- **使用摊薄股份**：包括期权、RSU、可转换证券
- **其他调整**（如适用）：
  - 少数股东权益
  - 养老金负债
  - 经营租赁义务

**估值输出格式：**
```csv
估值组成部分，金额（$百万）
预测期FCF现值，X.X
终值现值，Y.Y
企业价值，Z.Z
(-) 净债务，A.A
股权价值，B.B
,,
流通股数（百万），C.C
隐含每股价格，$XX.XX
当前股价，$YY.YY
隐含上升/(下降)，+XX%
```

### 步骤10：敏感性分析

在DCF表的底部构建**三个敏感性表**，显示估值如何随不同假设变化：

1. **WACC vs 终值增长** - 显示企业价值对折现率和永续增长的敏感性
2. **收入增长 vs EBIT利润率** - 显示顶线增长和运营杠杆的影响
3. **贝塔 vs 无风险利率** - 显示对股权成本组成部分的敏感性

**实现**：这些是简单的2D网格（不是Excel的"数据表"功能），每个单元格都有公式。每个单元格必须包含针对该特定假设组合的完整DCF重新计算。有关使用openpyxl以编程方式填充所有75个单元格的详细要求，请参阅关键约束部分。

<correct_patterns>

本部分包含构建DCF模型时应遵循的所有正确模式。

### 情景块选择模式 - 遵循此方法

**假设在每个情景的单独块中组织：**

**关键结构 - 每个部分标题三行：**

```csv
悲观情景假设（部分标题，合并跨单元格）
假设，FY1，FY2，FY3，FY4，FY5
收入增长率（%），12%，10%，9%，8%，7%
EBIT利润率（%），45%，44%，43%，42%，41%

基准情景假设（部分标题，合并跨单元格）
假设，FY1，FY2，FY3，FY4，FY5
收入增长率（%），16%，14%，12%，10%，9%
EBIT利润率（%），48%，49%，50%，51%，52%

乐观情景假设（部分标题，合并跨单元格）
假设，FY1，FY2，FY3，FY4，FY5
收入增长率（%），20%，18%，15%，13%，11%
EBIT利润率（%），50%，51%，52%，53%，54%
```

**每个情景块必须有一个列标题行**，紧接部分标题下方显示预测年度（FY2025E、FY2026E等）。没有这个，用户无法知道哪个假设值对应哪一年。

**如何引用假设 - 创建合并列：**
1. 情景选择器单元格（例如，B6）包含1=悲观、2=基准或3=乐观
2. 创建一个带有INDEX或OFFSET公式的合并列，以从正确的情景块中提取
3. 预测公式引用合并列（干净的单元格引用）
4. 每个情景块包含跨预测年度的完整DCF假设集

**推荐的合并列模式（使用INDEX）：**
`=INDEX(B10:D10, 1, $B$6)`

**不要这样 - 到处都是分散的IF语句：**
`=IF($B$6=1,[悲观块单元格],IF($B$6=2,[基准块单元格],[乐观块单元格]))`

合并列方法集中了逻辑，使模型更容易审计。

### 正确的收入预测模式

**创建一个带有INDEX公式的合并列，然后在预测中引用它：**

**步骤1 - FY1增长的合并列：**
`=INDEX([悲观FY1增长]:[乐观FY1增长], 1, $B$6)`

**步骤2 - 收入预测引用合并列：**
`收入第1年：=D29*(1+$E$10)`

其中：
- D29 = 前一年收入
- $E$10 = FY1增长的合并列单元格（包含INDEX公式）
- $B$6 = 情景选择器（1=悲观、2=基准、3=乐观）

**这种方法比在每个预测公式中嵌入IF语句更清晰**，并且使审计正在使用哪些情景假设变得更容易。

### 正确的FCF公式模式

**使用带有INDEX公式的合并列，然后在FCF计算中引用它们：**

**合并列方法：**
```csv
项目，公式，引用
D&A，=E29*$E$21，$E$21 = D&A %的合并列
CapEx，=E29*$E$22，$E$22 = CapEx %的合并列
Δ NWC，=(E29-D29)*$E$23，$E$23 = NWC %的合并列
无杠杆FCF，=E57+E58-E60-E62，E57=NOPAT E58=D&A E60=CapEx E62=Δ NWC
```

**每个合并列单元格包含一个INDEX公式**，根据情景选择器从相应的情景块中提取。这使预测公式干净且可审计。

在编写公式之前，确认情景块行位置并设置合并列。

1. **部分标题行**（合并单元格）：例如，"悲观情景假设"

### 正确的敏感性表实现

**重要**：这些不是Excel的"数据表"功能。这些是简单的网格，您使用openpyxl在其中编写常规公式。是的，这意味着总共约75个公式（3个表 × 每个表25个单元格），但这很直接且是必需的。

**使用公式以编程方式填充：**

每个敏感性表必须完全填充公式，为每个假设组合重新计算隐含股价。**不要使用Excel的数据表功能**（它需要手动干预，无法通过openpyxl自动化）。

**实现方法 - 具体示例：**

**表结构 - 5×5网格（奇数维度，基准情景居中）：**

如果模型的基准WACC = 9.0%且基准终值增长 = 3.0%，围绕这些值对称地构建轴：

```csv
WACC vs 终值增长，  2.0%，  2.5%，  3.0%，  3.5%，  4.0%
              8.0%，       [公式]，[公式]，[公式]，[公式]，[公式]
              8.5%，       [公式]，[公式]，[公式]，[公式]，[公式]
              9.0%，       [公式]，[公式]，[★  ]，[公式]，[公式]   ← 中间行 = 基准WACC
              9.5%，       [公式]，[公式]，[公式]，[公式]，[公式]
             10.0%，       [公式]，[公式]，[公式]，[公式]，[公式]
                                   ↑
                          中间列 = 基准终值g
```

**★ = 中心单元格。** 其公式输出必须等于模型的实际隐含股价（来自估值摘要）。对此单元格应用中等蓝色填充（`#BDD7EE`）和粗体字体，以便基准情景在视觉上锚定。

**轴值规则**：`轴值 = [基准 - 2*步长，基准 - 步长，基准，基准 + 步长，基准 + 2*步长]` - 围绕基准对称，奇数计数保证中心。

**公式模式 - 单元格B88（WACC=8.0%，终值增长=2.0%）：**

B88中的公式应使用以下内容重新计算隐含价格：
- 来自行标题的WACC：`$A88`（8.0%）
- 来自列标题的终值增长：`B$87`（2.0%）

**推荐方法**：引用主DCF计算，但替换这些值。

**示例公式结构：**
`=([使用$A88作为折现率的FCF现值之和] + [使用B$87作为增长率和$A88作为WACC的终值] - [净债务]) / [股份]`

**关键 - 在5x5网格的每个单元格编写一个公式（每个表25个单元格，总共75个单元格）。** 使用openpyxl在循环中以编程方式编写这些公式。不要跳过此步骤或留下占位符文本。

**Python实现模式：**
```python
# 填充敏感性表的伪代码
for row_idx, wacc_value in enumerate(wacc_range):
    for col_idx, term_growth_value in enumerate(term_growth_range):
        # 构建使用wacc_value和term_growth_value的公式
        formula = f"=<使用{wacc_value}和{term_growth_value}的DCF重新计算>"
        ws.cell(row=start_row+row_idx, column=start_col+col_idx).value = formula
```

**敏感性表必须在用户打开模型时立即工作，不需要用户进行任何手动步骤。**

<common_mistakes>

本部分包含构建DCF模型时应避免的所有错误模式。

### 错误：简化的敏感性表近似或占位符文本

**不要使用线性近似：**

```
// 错误 - 线性近似
B97: =B88*(1+(0.096-0.116))    // 假设线性关系

// 错误 - 除法捷径
B105: =B88/(1+(E48-0.07))      // 不重新计算完整DCF
```

**不要留下占位符文本：**
```
// 错误 - 占位符注释
"注意：使用Excel数据表功能（数据 → 假设分析 → 数据表）来填充敏感性表。"

// 错误 - 空单元格
[因为"这很复杂"而留空单元格]
```

**不要混淆术语：**
- ❌ "敏感性表需要Excel的数据表功能"（不 - 那是我们要使用的特定Excel工具）
- ✅ "敏感性表是每个单元格都有公式的简单网格"（是 - 这就是我们构建的）

**为什么这些捷径是错误的：**
- 线性近似公式实际上不重新计算DCF - 它们只是应用简单的数学调整
- 关系不是线性的，所以结果会不准确
- 占位符文本需要用户手动干预
- 交付时模型不能立即使用
- 不专业或客户就绪
- 空单元格 = 未完成的交付物

**要拒绝的常见理由：**
"编写75+个公式感觉很复杂，所以我留一个注释让用户手动完成。"

**现实**：当你使用Python的openpyxl中的循环时，编写75个公式很简单。每个公式遵循相同的模式 - 只是替换行/列值。这是交付物的必需部分。

**相反**：使用公式填充每个敏感性单元格，为该特定假设组合重新计算完整的DCF

### 错误：缺少单元格注释

**不要这样做：**
- 在没有注释的情况下创建所有硬编码输入
- 想"我稍后会添加它们"
- 写"TODO：添加来源"
- 留下蓝色输入没有文档

**为什么它是错误的：**
- 无法验证数据来自哪里
- 不符合xlsx技能要求
- 不具备审计准备
- 浪费时间以后修复

**相反**：在创建每个硬编码值时添加单元格注释

### 错误：公式行引用错误

**症状：**
FCF部分引用错误的假设行：
`D&A：  =E29*$E$34    // 应该是$E$21，但引用了错误的行`
`CapEx： =E29*$E$41   // 应该是$E$22，但行偏移了`

**为什么发生这种情况：**
1. 首先编写公式
2. 然后插入标题
3. 所有行引用偏移
4. 现在公式指向错误的单元格 → #REF!错误

**相反**：首先锁定行布局，然后编写公式

### 错误：每个情景中假设只有一行

**不要这样构建假设：**
```csv
假设，悲观，基准，乐观
收入增长率FY1，10%，13%，16%
收入增长率FY2，9%，12%，15%
```
这种垂直布局很难看到每个情景中跨年度的进展。

**为什么它是错误的：**
- 难以看到每个情景中跨年度演变的假设
- 更难跨完整预测期比较情景假设
- 审查情景逻辑时不直观

**相反：**
- 为每个情景创建单独的块（悲观、基准、乐观）
- 在每个块内，横向显示跨预测年度的假设
- 这使每个情景的假设更容易作为一组连贯的集合进行审查

### 错误：无边框

**不要交付没有边框的模型：**
- 没有部分划分
- 所有单元格混合在一起
- 难以阅读且不专业

**为什么它是错误的：**
- 不具备客户就绪性
- 难以导航
- 看起来业余

**相反**：在所有主要部分周围添加边框

### 错误：错误的字体颜色或没有字体颜色区分

**不要这样做：**
- 所有文本都是黑色
- 只使用填充颜色（没有字体颜色变化）
- 混淆哪些单元格是蓝色还是黑色

**为什么它是错误的：**
- 无法区分输入和公式
- 审计变得不可能
- 违反xlsx技能要求

**相反**：所有硬编码输入使用蓝色文本，所有公式使用黑色文本，工作表链接使用绿色

### 错误：运营费用基于毛利润

**不要这样做：**
`S&M： =E33*0.15    // E33 = 毛利润（错误）`

**为什么它是错误的：**
- 运营费用随收入扩展，而不是毛利润
- 产生不切实际的利润率进展
- 不是企业实际运营的方式

**相反：**
`S&M： =E29*0.15    // E29 = 收入（正确）`

### 前5个错误摘要

1. **公式行引用错误** → 在编写公式之前定义所有行位置
2. **缺少单元格注释** → 在创建单元格时添加注释，而不是在最后
3. **简化的敏感性表** → 使用完整的DCF重新计算公式填充所有单元格，而不是近似
4. **情景块引用错误** → 确保IF公式从正确的悲观/基准/乐观块中提取
5. **无边框** → 为客户就绪的外观添加专业部分边框

此外，请注意这些错误：

### WACC计算错误
- 在资本结构中混合账面价值和市场价值
- 错误地使用股权贝塔而不是资产/无杠杆贝塔
- 对债务成本应用错误的税率
- 错误的无风险利率（必须使用当前10年期国债）
- 未能调整净债务与净现金头寸

### 增长假设缺陷
- 终值增长 > WACC（产生无限价值）
- 预测增长率与历史表现不一致
- 忽略行业增长约束
- 收入增长与单元经济不一致
- 没有运营理由的利润率扩展

### 终值错误
- 使用错误的增长方法（永续 vs 退出倍数）
- 终值 > 企业价值的80%（表明过度依赖）
- 与稳态假设的终值利润率不一致
- 终值的错误折现期

### 现金流预测错误
- 运营费用基于毛利润而不是收入
- D&A/CapEx百分比与业务模式不一致
- 营运资本变化计算不正确
- 年度间税率不一致
- NOPAT计算错误

**这些错误是最常见的。在开始任何DCF构建之前重新阅读本部分。**

## Excel模型结构

### 表格架构

创建**两个工作表**：

1. **DCF** - 主估值模型，底部有敏感性分析
2. **WACC** - 资本成本计算

**关键**：敏感性表放在DCF表的底部（不是单独的工作表）。这将所有估值输出保持在一起。

### 公式重新计算（必需）

创建或修改Excel模型后，**使用xlsx技能的recalc.py脚本重新计算所有公式**：

```bash
python recalc.py [excel文件路径] [超时秒数]
```

示例：
```bash
python recalc.py AAPL_DCF_Model_2025-10-12.xlsx 30
```

脚本将：
- 使用LibreOffice重新计算所有工作表中的所有公式
- 扫描所有单元格的Excel错误（#REF!、#DIV/0!、#VALUE!、#NAME?、#NULL!、#NUM!、#N/A）
- 返回详细的JSON，包含错误位置和计数

**预期输出格式：**
```json
{
  "status": "success",           // 或 "errors_found"
  "total_errors": 0,              // 总错误计数
  "total_formulas": 42,           // 文件中的公式数
  "error_summary": {}             // 仅在发现错误时存在
}
```

**如果发现错误**，输出将包含详细信息：
```json
{
  "status": "errors_found",
  "total_errors": 2,
  "total_formulas": 42,
  "error_summary": {
    "#REF!": {
      "count": 2,
      "locations": ["DCF!B25", "DCF!C25"]
    }
  }
}
```

**修复所有错误**并重新运行recalc.py，直到状态为"success"后再交付模型。

### 格式标准

**重要**：遵循xlsx技能的公式构建规则和数字格式约定。DCF技能添加了特定的视觉呈现标准。

**配色方案 - 两层：**

**第1层：字体颜色（xlsx技能必需）**
- **蓝色文本（RGB：0,0,255）**：所有硬编码输入（股价、股份、历史数据、假设）
- **黑色文本（RGB：0,0,0）**：所有公式和计算
- **绿色文本（RGB：0,128,0）**：指向其他工作表的链接（WACC工作表引用）

**第2层：填充颜色 - 专业蓝/灰色调（默认，除非用户另有说明）**
- **保持最小化** - 仅使用蓝色和灰色进行填充。不要引入绿色、黄色、橙色或多种强调色。颜色太多的模型看起来业余。
- **默认填充调色板：**
  - **部分标题**：深蓝色（RGB：31,78,121 / `#1F4E79`）背景配白色粗体文本
  - **子标题/列标题**：浅蓝色（RGB：217,225,242 / `#D9E1F2`）背景配黑色粗体文本
  - **输入单元格**：浅灰色（RGB：242,242,242 / `#F2F2F2`）背景配蓝色字体 - 或者如果需要最大最小化，只需白色配蓝色字体
  - **计算单元格**：白色背景配黑色字体
  - **输出/摘要行**（每股价值、EV等）：中等蓝色（RGB：189,215,238 / `#BDD7EE`）背景配黑色粗体字体
  - **就是这样 - 3种蓝色 + 1种灰色 + 白色。** 抵制添加更多颜色的冲动。
  - 用户提供的模板或明确的颜色偏好始终覆盖这些默认值。

**各层如何协同工作：**
- 输入单元格：蓝色字体 + 浅灰色填充 = "硬编码输入"
- 公式单元格：黑色字体 + 白色背景 = "计算值"
- 工作表链接：绿色字体 + 白色背景 = "来自另一个工作表的引用"
- 关键输出：黑色粗体字体 + 中等蓝色填充 = "这就是答案"

**字体颜色告诉你它是什么（输入/公式/链接）。填充颜色告诉你你在哪里（标题/数据/输出）。**

### 边框标准（专业外观必需）

**厚边框**（1.5pt）围绕主要部分：
- 关键输入部分
- 预测假设部分
- 5年现金流预测部分
- 终值部分
- 估值摘要部分
- 每个敏感性分析表

**中等边框**（1pt）在子部分之间：
- 公司详情 vs 历史表现
- 增长假设 vs EBIT利润率 vs FCF参数

**薄边框**（0.5pt）围绕数据表：
- 情景假设表（悲观 | 基准 | 乐观 | 已选）
- 历史vs 预测财务矩阵

**无边框**：表格内的单个单元格（保持干净、可扫描）

**边框是必需的** - 没有专业边框的模型不具备客户就绪性。

**数字格式**（遵循xlsx技能标准）：
- **年份**：格式化为文本字符串（例如，"2024"而不是"2,024"）
- **百分比**：`0.0%`（一位小数）
- **货币**：`$#,##0`用于百万；`$#,##0.00`用于每股 - 始终在标题中指定单位（"收入（$mm)"）
- **零**：使用数字格式使所有零为"-"（例如，`$#,##0;($#,##0);-`）
- **大数字**：`#,##0`带千位分隔符
- **负数**：`(#,##0)`括号内（不是减号）

**单元格注释（所有硬编码输入必需）**：

根据xlsx技能，所有硬编码值必须具有记录来源的单元格注释。格式："来源：[系统/文档]，[日期]，[参考]，[适用时的URL]"

**关键**：在创建单元格时添加注释。不要推迟到最后。

### 敏感性分析（DCF工作表底部）

**术语提醒**："敏感性表" = 具有行标题、列标题和每个数据单元格中公式的简单2D网格。不是Excel的"数据表"功能（数据 → 假设分析 → 数据表）。您将使用openpyxl在每个单元格中编写常规Excel公式。

**位置**：DCF工作表的第87+行（不是单独的工作表）

**三个敏感性表，垂直堆叠：**

1. **WACC vs 终值增长**（第87-100行）- 5x5网格 = 25个带公式的单元格
2. **收入增长 vs EBIT利润率**（第102-115行）- 5x5网格 = 25个带公式的单元格
3. **贝塔 vs 无风险利率**（第117-130行）- 5x5网格 = 25个带公式的单元格

**要编写的公式总数：75**（这是必需的，不是可选的）

**关键**：所有敏感性表单元格必须使用openpyxl以编程方式填充公式。不要使用线性近似捷径。不要留下占位符文本或关于手动步骤的注释。不要因为"它很复杂"而合理化留空单元格 - 使用Python循环生成公式。

**表设置：**
1. 创建具有行/列标题的表结构（要测试的假设值）
2. 用以下内容填充每个数据单元格的公式：
   - 使用行标题值（例如，WACC = 9.0%）
   - 使用列标题值（例如，终值增长 = 3.0%）
   - 使用这些特定假设重新计算完整的DCF

6. 在表之间留1-2个空行

**不需要手动干预** - 用户打开文件时敏感性表必须完全可用。

## 情景选择器实现

**三情景框架：**

### 悲观情景
- 保守的收入增长（历史范围的低端）
- 利润率压缩或无扩展
- 更高的WACC（风险溢价增加）
- 更低的终值增长率
- 更高的CapEx假设

### 基准情景
- 共识或管理层指导的收入增长
- 基于运营杠杆的适度利润率扩展
- 当前市场隐含的WACC
- 与GDP一致的终值增长（2.5-3.0%）
- 标准CapEx假设

### 乐观情景
- 乐观的收入增长（预测的高端）
- 显著的利润率扩展
- 更低的WACC（降低风险溢价）
- 更高的终值增长（3.5-5.0%）
- 降低CapEx强度

**公式实现：**

**不要使用到处都是的嵌套IF公式。** 相反，创建一个使用INDEX或OFFSET公式从适当情景块中提取的合并列。

**推荐模式（使用INDEX）：**
`=INDEX(B10:D10, 1, $B$6)`，其中`B10:D10` = 悲观/基准/乐观值，`1` = 行偏移，`$B$6` = 情景选择器单元格（1、2或3）

**然后在所有预测中引用合并列**：
`收入第1年：=D29*(1+$E$10)`，其中$E$10是第1年增长的合并列值。

这种方法集中了情景逻辑，使模型更容易审计和维护。

## 常见变体

### 高增长科技公司
- 更长的预测期（7-10年）
- 更高的初始增长率（20-30%）
- 随时间推移的显著利润率扩展
- 更高的WACC（12-15%）
- 建模单元经济（用户、ARPU等）

### 成熟/稳定公司
- 更短的预测期（3-5年）
- 适度的增长率（GDP +1-3%）
- 稳定的利润率
- 更低的WACC（7-9%）
- 专注于现金产生和资本配置

### 周期性公司
- 通过经济周期建模
- 在周期中期正常化利润率
- 考虑低谷和峰值情景
- 针对周期性调整贝塔

### 多段公司
- 为每个业务单元创建单独的DCF
- 按分部不同的增长率和利润率
- 各部分估值之和
- 考虑协同效应
